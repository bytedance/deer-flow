# 记忆系统改进

===================
设计思路说明
===================

**为什么需要记忆系统**：
1. 个性化对话：记住用户偏好和历史
2. 上下文连续性：跨会话保持信息
3. 智能检索：根据相关性选择记忆内容

**核心设计原则**：
- Token预算：控制注入量，避免超过模型限制
- 置信度排序：优先使用高置信度事实
- 未来扩展：支持TF-IDF相似度检索

**为什么使用tiktoken**：
- 准确计算token数
- 避免超出模型限制
- 优化token使用效率

本文档跟踪记忆注入行为和路线图状态。

## 当前行为

当前函数：


**当前注入格式**：
- `User Context` 部分，来自 `user.*.summary`
- `History` 部分，来自 `history.*.summary`
- `Facts` 部分，来自 `facts[]`，按置信度排序，追加直到达到 token 预算

**为什么这样组织**：
- User Context：用户的固定属性和偏好
- History：时间递减的历史上下文
- Facts：动态累积的事实性信息

**Token 计数**：
- 当可用时使用 `tiktoken` (`cl100k_base`)
- 如果 tokenizer 导入失败，回退到 `len(text) // 4`

**为什么有回退机制**：
- tiktoken可能未安装
- 避免因缺少依赖而失败
- 粗略估算总比没有好

## 路线图（计划中）

**计划评分策略**：

```text
final_score = (similarity * 0.6) + (confidence * 0.4)
```

**为什么这样分配权重**：
- 相似度更重要（0.6）：确保相关性
- 置信度次之（0.4）：保证信息质量
- 可根据实际效果调整

**计划集成形状**：
1. 从过滤的用户/最终助手轮次提取最近的对话上下文。
2. 计算每个事实与当前上下文之间的 TF-IDF 余弦相似度。
3. 按加权分数排序并在 token 预算下注入。
4. 如果上下文不可用，回退到仅置信度排序。

**为什么需要上下文感知**：
- 当前对话可能只与部分事实相关
- 减少无关信息的干扰
- 提高响应质量
