## 一、需求背景

DeerFlow 的 Local Sandbox 模式下，每个对话（thread）有独立的数据目录，隔离了工作区。但项目需要一个**所有对话共享的文件夹**，用于存放公共资源（如项目模板、共享素材、全局配置等）。

---

## 二、系统调研结论

### 官方文档情况

查阅 `backend/docs/CONFIGURATION.md` 发现：
- `sandbox.mounts` 配置**只在 Docker Sandbox 部分有文档说明**
- Local Sandbox 部分**没有提到 mounts 配置**
- 但代码中 `LocalSandboxProvider` 实际已包含读取 mounts 的逻辑（可能是社区贡献或未文档化的功能）

### 架构分析

Local Sandbox 的架构**天然支持**共享文件夹：
1. `LocalSandbox` 采用**单例模式**，所有对话共享同一个实例
2. `LocalSandboxProvider._setup_path_mappings()` 已能读取 `sandbox.mounts` 配置
3. `LocalSandbox._resolve_path()` 可自动将虚拟路径映射到宿主机路径
4. 工具函数中已有 `_is_custom_mount_path()` 验证逻辑

## 三、修改清单

### 1. 代码修改

#### 1.1 `LocalSandboxProvider._setup_path_mappings()`

**文件**: `backend/packages/harness/deerflow/sandbox/local/local_sandbox_provider.py`

**修改**: 支持相对路径自动解析为绝对路径

```python
# 新增相对路径解析逻辑
if not host_path.is_absolute():
    # base_dir is backend/.deer-flow, so .parent.parent is project root
    project_root = get_paths().base_dir.parent.parent
    host_path = (project_root / host_path).resolve()
```

**关键**: 添加了 `get_paths` 导入，使相对路径能正确解析到项目根目录。

#### 1.2 `_get_custom_mounts()`

**文件**: `backend/packages/harness/deerflow/sandbox/tools.py`

**修改**: 路径验证时支持相对路径解析

```python
# 新增相对路径解析逻辑
for m in config.sandbox.mounts:
    host_path = Path(m.host_path)
    if not host_path.is_absolute():
        host_path = (project_root / host_path).resolve()
    if host_path.exists():
        mounts.append(m)
```

**关键**: 原代码用 `Path(m.host_path).exists()` 直接检查，相对路径会相对于当前工作目录判断，导致误判为不存在。修复后先解析为绝对路径再检查。

### 2. 配置文件修改

#### 2.1 `config.yaml`

**文件**: `c:\xiangmu\deer-flow\config.yaml`

**新增 sandbox.mounts 配置**:

```yaml
sandbox:
  use: deerflow.sandbox.local:LocalSandboxProvider
  allow_host_bash: false
  mounts:
    - host_path: backend/.deer-flow/shared-data    # 主机路径（相对于项目根目录）
      container_path: /mnt/shared-data             # 沙箱内的虚拟路径
      read_only: false                             # 允许所有对话读写
```

**路径说明**：
- `host_path` 使用相对路径，相对于**项目根目录**（`deer-flow/`）
- 代码中会自动解析：`project_root = get_paths().base_dir.parent.parent`
- 解析链条：`backend/packages/harness/deerflow/config/` → `parents[4]` → `backend/` → `base_dir = backend/.deer-flow` → `.parent.parent` → 项目根目录
- 所以 `backend/.deer-flow/shared-data` 最终解析为 `c:\xiangmu\deer-flow\backend\.deer-flow\shared-data`
- **无论当前工作目录是哪个目录，解析结果始终一致**

## 四、工作原理

### 完整调用链路

```
Agent 调用 write_file("/mnt/shared-data/test.txt")
    ↓
write_file_tool()
    ↓
validate_local_tool_path(path, thread_data)
    ↓
_is_custom_mount_path(path)
    ↓
_get_custom_mounts() — 从 config.yaml 读取 mounts 配置
    ↓ [验证通过]
LocalSandbox.write_file(path, content)
    ↓
LocalSandbox._resolve_path(path)
    ↓ [路径映射]
/mnt/shared-data/test.txt → c:\xiangmu\deer-flow\backend\.deer-flow\shared-data\test.txt
    ↓
写入宿主机文件系统 ✅
```

### 关键技术点

1. **路径验证与路径解析是两套独立系统**:
   - 验证: `_get_custom_mounts()` → 用于安全校验（是否允许访问）
   - 解析: `LocalSandbox._resolve_path()` → 用于实际读写（虚拟路径→宿主机路径）
   - 两者必须都包含挂载配置，缺一不可

2. **单例模式**: `LocalSandbox` 是全局单例，所有对话共享同一个实例和路径映射表

3. **相对路径**: 配置中使用相对路径 `backend/.deer-flow/shared-data`，代码自动相对于项目根目录解析

## 五、问题修复记录

### 问题1: 写入文件 Permission denied

- **现象**: 能列出 `/mnt/shared-data`（显示为空），但写入时报 Permission denied
- **原因**: `_get_custom_mounts()` 检查路径是否存在时，没有先解析相对路径。`Path("backend/.deer-flow/shared-data").exists()` 相对于当前工作目录判断为不存在，导致挂载被过滤掉
- **修复**: 在 `_get_custom_mounts()` 中添加相对路径解析逻辑，与 `LocalSandboxProvider._setup_path_mappings()` 保持一致

### 问题2: 配置了 mounts 但不生效

- **现象**: `config.yaml` 中写了 mounts，但 agent 无法访问
- **原因**: `LocalSandboxProvider` 原代码要求 `host_path` 必须是绝对路径，否则会跳过该挂载
- **修复**: 添加相对路径自动解析逻辑，相对于项目根目录解析

## 七、使用指南

### Agent 调用示例

```yaml
# 在 Agent 的 SOUL.md 或 prompt 中说明：

## 共享数据目录

所有对话共享的数据存放在 `/mnt/shared-data` 目录中：
- 公共模板、参考文档
- 跨对话共享的资源
- 全局配置文件

使用方式：
- 读取: `read_file("/mnt/shared-data/templates/novel-template.md")`
- 写入: `write_file("/mnt/shared-data/shared-notes.md", "内容")`
- 列出: `ls("/mnt/shared-data")`
```

### 目录结构建议

```
backend/.deer-flow/shared-data/
├── templates/        # 项目模板
├── references/       # 参考资料
├── assets/           # 共享素材
── config/           # 全局配置
```

## 八、注意事项

1. **配置必须**: `config.yaml` 中的 `sandbox.mounts` 是必须配置的，否则无法使用共享文件夹
2. **重启生效**: 修改配置后需要 `make stop && make dev` 重启服务
3. **权限**: `read_only: false` 允许所有对话读写共享目录
4. **路径安全**: 共享目录内的路径遍历检查与其他目录一样生效
5. **Docker Sandbox**: Docker 模式下 mounts 配置同样适用，由 `AioSandboxProvider` 处理
