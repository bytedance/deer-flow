# 全局变量动态提示词注入需求文档

## 1. 业务需求

### 1.1 需求背景

当前项目中，大模型的提示词（Prompt）在每次对话时是相对静态的，主要由以下几个部分组成：
- 角色定义（role）
- 灵魂文件（soul）
- 记忆上下文（memory_context）
- 技能部分（skills_section）
- 子代理部分（subagent_section）
- 工作目录（working_directory）
- 响应风格（response_style）
- 引用规范（citations）
- 关键提醒（critical_reminders）

**业务痛点**：
- 无法在对话过程中动态调整提示词内容
- 业务脚本无法实时影响大模型的行为
- 每次调用大模型时，提示词是固定的（如 a、b、c、d），无法根据上下文动态变化（如变成 a、b、c、f）

### 1.2 需求目标

实现一个**全局变量系统**，支持：

1. **变量存储**：维护一组全局变量，可被系统读取和修改
2. **工具调用**：大模型可以通过工具修改这些全局变量
3. **业务脚本**：外部业务脚本可以通过 API 修改全局变量
4. **动态注入**：每次调用大模型时，将全局变量作为动态提示词注入到系统提示词中

### 1.3 使用场景示例

```
场景1：动态工作模式切换
- 初始提示词：a、b、c、d（包含全局变量 mode=normal）
- 用户说"切换到写作模式"
- 大模型调用工具修改全局变量 mode=writing
- 下次调用时提示词变为：a、b、c、f（包含全局变量 mode=writing）

场景2：业务脚本动态注入
- 外部脚本检测到某个业务状态变化
- 脚本通过 API 修改全局变量 current_task="审核章节"
- 大模型下次响应时，会根据新的上下文调整行为

场景3：多阶段任务
- 第一次调用：阶段=规划，提示词包含规划相关的指令
- 第二次调用：阶段=执行，提示词包含执行相关的指令
- 第三次调用：阶段=审核，提示词包含审核相关的指令
```

| Soul | `get_agent_soul()` | 每次调用 | ✅ 完全复用 |
| Deferred Tools | `get_deferred_tools_prompt_section()` | 每次调用 | ✅ 完全复用 |

**关键验证**：
- `apply_prompt_template()` 在**每次大模型调用时**都会被调用
- 所有注入内容都是**实时从存储读取**的，不是缓存的
- 使用 Python 的 `.format()` 方法进行变量替换，简单可靠

#### 2.1.2 工具系统可复用

项目已有成熟的工具注册机制（参考 `tools/tools.py` 和 `skill_manage_tool.py`），全局变量工具可以：
- 复用现有的工具注册流程
- 复用工具权限控制
- 复用工具调用响应机制

#### 2.1.3 存储机制可复用

项目的 Memory 系统已实现文件存储 + 缓存机制（`memory/storage.py`），全局变量可以：
- 复用文件锁保证并发安全
- 复用缓存机制提升性能
- 复用数据持久化流程

### 2.2 变量生效范围：两级隔离

#### 2.2.1 设计决策

根据你的要求，全局变量系统支持**两级隔离**：

| 隔离级别 | 生效范围 | 使用场景 |
|----------|----------|----------|
| **项目级** | 所有对话共享 | 项目配置、全局规则、工作模式等 |
| **对话级** | 仅该对话使用 | 当前章节号、临时状态、对话上下文等 |

**为什么不需要 Agent 级？**
- 项目级已经可以让所有 Agent 共享配置
- 对话级可以让每个对话独立控制状态
- 去掉 Agent 级可以简化设计，降低维护成本

#### 2.2.2 覆盖规则

**对话级 > 项目级**

同名变量时，对话级变量优先覆盖项目级变量。

#### 2.2.3 生效示例

```
场景1：项目级变量
- 设置：mode="writing"（项目级）
- 生效：所有对话都使用 mode="writing"

场景2：对话级变量
- 设置：current_chapter=5（对话级，针对对话A）
- 生效：只有对话A使用 current_chapter=5
- 其他对话不受影响

场景3：覆盖场景
- 项目级：mode="writing"
- 对话A设置：mode="reviewing"（对话级）
- 结果：对话A使用 reviewing，其他对话仍用 writing
```

## 3. 配置说明

### 3.1 配置文件位置

配置文件在项目根目录的 `config.yaml`（读音：config dot Y-A-M-L）。

### 3.2 全局变量配置

在 `config.yaml` 中添加以下配置：

```yaml
# config.yaml - 全局变量系统配置
global_variables:
  enabled: true                    # 是否启用全局变量系统
  max_variables_per_scope: 50      # 每个级别最多多少个变量
  max_value_length: 500            # 单个变量值最多多少个字符
  max_total_prompt_length: 3000    # 注入到提示词的最大长度
```

**配置含义**：
- `enabled`：关掉就不注入，不影响现有功能
- `max_variables_per_scope`：项目级最多 50 个，对话级最多 50 个
- `max_value_length`：单个变量值太长会影响提示词，所以要限制
- `max_total_prompt_length`：所有变量加起来的长度限制

### 3.3 配置文件怎么改

1. 打开项目根目录下的 `config.yaml`
2. 找到合适的位置添加上面的配置
3. 重启服务生效

### 3.4 后续要修改的提示词文件

实现全局变量系统后，你可以在以下位置添加变量引用：

| 修改位置 | 文件 | 用途 |
|----------|------|------|
| Agent 灵魂文件 | `agents/{name}/SOUL.md` | 定义 Agent 的性格和行为 |
| Agent 配置 | `agents/{name}/config.yaml` | 定义 Agent 使用的工具 |
| 技能文件 | `skills/{name}/SKILL.md` | 定义技能的执行流程 |
| 工具配置 | `config.yaml` 的 tools 部分 | 定义工具的参数和行为 |

**如何引用变量**：

在提示词中直接使用变量名即可。例如在 `SOUL.md` 中：

```markdown
## 当前工作模式
当前工作模式为：{{mode}}

## 当前章节
正在处理第 {{current_chapter}} 章
```

系统会自动将 `{{变量名}}` 替换为对应的变量值。

    skills_section = get_skills_prompt_section(available_skills)
    deferred_tools_section = get_deferred_tools_prompt_section()
    acp_section = _build_acp_section()
    
    # 使用 Python 的 .format() 方法进行变量替换

        agent_name=agent_name or "DeerFlow 2.0",
        soul=get_agent_soul(agent_name),
        skills_section=skills_section,
        deferred_tools_section=deferred_tools_section,

## 5. 实现方案

### 4.1 方案概述

基于现有架构，采用**两级隔离**（项目级 + 对话级）的全局变量系统：

```
┌─────────────────┐
│   业务脚本       │  ← 通过 API 修改变量
└────────┬────────┘
         │
         ▼
┌─────────────────┐     ┌──────────────────┐
│   变量存储       │     │  项目级变量文件   │ ← 所有对话共享
│   (两级隔离)     │     │  对话级变量文件   │ ← 每个对话独立
└────────┬────────┘     └──────────────────┘
         │
         ▼
┌─────────────────┐
│   大模型工具     │  ← 对话中修改变量
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  提示词注入      │  ← 每次调用自动合并两级变量
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  大模型收到      │
│  带变量的提示词   │
└─────────────────┘
```

### 4.2 存储位置

| 级别 | 存储位置 | 说明 |
|------|----------|------|
| 项目级 | `{DEER_FLOW_HOME}/global_variables.json` | 所有对话共享 |
| 对话级 | `{DEER_FLOW_HOME}/threads/{thread_id}/variables.json` | 每个对话独立 |

### 4.3 数据结构

每个变量包含以下信息：

```json
{
  "variables": {
    "mode": {
      "value": "writing",
      "description": "当前工作模式",
      "updated_at": "2026-04-27T10:00:00Z",
      "updated_by": "llm"
    },
    "current_chapter": {
      "value": "5",
      "description": "当前章节号",
      "updated_at": "2026-04-27T10:05:00Z",
      "updated_by": "llm"
    }
  }
}
```

### 4.4 大模型可用的工具

实现后，大模型会自动获得以下工具：

| 工具名 | 用途 | 示例 |
|--------|------|------|
| `get_variable(key, scope)` | 获取变量值 | 获取 `mode` 的值 |
| `set_variable(key, value, scope)` | 设置变量 | 设置 `mode=writing` |
| `delete_variable(key, scope)` | 删除变量 | 删除 `mode` |
| `list_variables(scope)` | 列出所有变量 | 列出当前所有变量 |

**scope 参数**：
- `project`：操作项目级变量
- `thread`：操作当前对话级变量

### 4.5 提示词注入方式

每次大模型被调用时，系统会自动：

1. 读取项目级变量
2. 读取当前对话级变量
3. 对话级变量覆盖项目级同名变量
4. 合并后注入到提示词中

**注入位置**：在系统提示词的 `memory`（记忆）之后，`skills`（技能）之前。

**注入格式**：

```
<global_variables>
- mode = writing  # 当前工作模式
- current_chapter = 5  # 当前章节号
- project_name = 我的小说  # 项目名称
</global_variables>
```

| 文件 | 用途 |

| `backend/packages/harness/deerflow/global_variables/storage.py` | 变量存储读写 |

### 4.3 方案优势

| 优势 | 说明 |
|------|------|
| **轻量级** | 基于 JSON 文件，无需额外数据库 |
| **易集成** | 与现有架构高度一致，复用现有注入机制 |
| **实时性** | 每次调用大模型时动态读取，确保最新 |
| **可扩展** | 后续可轻松升级为数据库存储 |
| **线程安全** | 使用锁机制保证并发安全 |

### 4.4 技术约束

| 约束项 | 说明 |
|--------|------|
| **变量数量** | 建议每个级别不超过 50 个，避免提示词过长 |
| **变量值长度** | 单个变量值建议不超过 500 字符 |
| **并发控制** | 使用文件锁保证并发安全 |
| **版本兼容** | 存储格式需支持未来扩展 |
| **覆盖规则** | 对话级 > 项目级 |
