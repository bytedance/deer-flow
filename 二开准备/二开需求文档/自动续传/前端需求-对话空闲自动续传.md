# 前端需求 - 对话空闲自动续传

> 为小说创作系统提供整夜挂机自动续传功能，前端需增加监控开关和状态显示。

---

## 1. 业务需求

### 1.1 背景

用户在整夜挂机执行写正文流程时，需要一种机制自动检测并续传任务，无需人工干预。

### 1.2 核心需求

1. **监控开关**：对话页面右侧增加监控按钮，开启后进入监控模式
2. **状态显示**：左侧会话列表显示监控中的对话（图标标识）
3. **自动判断**：根据 `card.json` 中的 `current_chapter` 和 `target_chapters` 判断是否进入监控
4. **空闲检测**：用户配置的空闲时间内无 LLM 输出时，自动发送续传消息
5. **可配置超时**：点击监控按钮后弹出配置面板，用户可自定义空闲超时时间

## 3. 交互逻辑

### 3.1 监控开关启用流程

```
用户点击监控按钮
   ↓
弹出配置面板
   ↓
用户设置空闲超时时间（分钟）
   ↓
点击"开始监控"
   ↓
关闭配置面板，更新按钮为"监控中"状态
   ↓
开始轮询：
  1. 读取 card.json（通过文件系统 API）
  2. 检查 current_chapter 和 target_chapters
  3. 如果 target_chapters > current_chapter → 进入监控状态
   ↓
更新 UI：
  - 监控按钮高亮 + 脉冲动画
  - 左侧列表显示监控图标
```

### 3.2 空闲检测逻辑

```
监控启用后：
  每 30 秒检查一次：
    1. thread.isLoading === false（Agent 空闲，run 已结束）
    2. card.json 中 target_chapters > current_chapter
    3. 空闲时长是否超过用户配置的超时时间
   ↓
条件全部满足 → 自动调用 sendMessage()：

续传提示词模板：
```
<system_notification>
当前是自动续传任务，触发条件：card.json 中 target_chapters（{target_chapters}）> current_chapter（{current_chapter}）。

请继续写第 {current_chapter + 1} 章，一直写到第 {target_chapters} 章。

⚠️ 重要：每完成一章后，请立即更新 card.json 中的 current_chapter 数值。
   - 写完第 N 章后，将 current_chapter 设为 N
   - 当 current_chapter >= target_chapters 时，表示全部完成，监控将自动停止
</system_notification>
```
   ↓
发起新 run：
  POST /api/threads/{thread_id}/runs/stream
  body: { input: { messages: [{type: "human", content: ...}] } }
  LangGraph checkpointer 自动加载历史，Agent 基于完整历史继续执行
   ↓
重置空闲计时器
```

### 3.3 防卡死机制（LLM 接口报错处理）

项目已具备 LLM 接口报错检测能力：
- `thread.error` 状态（来自 `useStream` hook）
- `llm_retry` 事件（后端 [llm_error_handling_middleware](file:///c:/xiangmu/deer-flow/backend/packages/harness/deerflow/agents/middlewares/llm_error_handling_middleware.py) 通过 `get_stream_writer()` 发送）
- `onError` 回调（[hooks.ts:289-292](file:///c:/xiangmu/deer-flow/frontend/src/core/threads/hooks.ts#L289-L292)）

**检测规则：**

```
每次 sendMessage() 后记录时间戳，监控以下模式：

"发送 → 停止 → 发送 → 停止 → 发送 → 停止"
  ↑                                    ↑
第一次续传                         第三次续传后停止

如果从第一次续传后 Agent 停止，到第三次续传后再次停止，
整个时间间隔 < 2 分钟 → 判定为 LLM 接口持续报错/卡死
```

**自动处理策略：**

```
第 1 阶段：正常监控
  └─ 检测到 3 次续传均在 2 分钟内停止
  └─ 判定为接口报错 → 自动暂停监控

第 2 阶段：10 分钟后重试
  └─ 暂停 10 分钟
  └─ 重新启用监控，尝试 1 次
  └─ 如果还是 2 分钟内停止 → 进入第 3 阶段

第 3 阶段：彻底停止
  └─ 重试失败 → 永久关闭监控
  └─ 显示 toast 提示用户："LLM 接口持续报错，监控已自动关闭"
```

### 3.4 监控关闭流程

```
用户再次点击监控按钮
   ↓
弹出确认提示（可选）
   ↓
停止轮询
   ↓
停止空闲检测
   ↓
更新 UI：
  - 监控按钮恢复默认状态
  - 左侧列表移除监控图标
```

### 4.5 监控逻辑实现

```typescript
// 监控主循环
function startMonitoring(threadId: string) {
  // 每 30 秒检查一次
  const checkTimer = setInterval(async () => {
    // 1. 检查是否启用监控
    if (!monitorEnabled) return
    
    // 2. 检查 thread 是否空闲
    const thread = useThreadStream.getState().getThread(threadId)
    if (thread?.isLoading) return  // LLM 正在输出，不触发
    
    // 3. 读取 card.json
    try {
      const card = await getNovelCard(threadId)
      
      // 4. 检查是否需要续传
      const shouldResume = card.target_chapters > card.current_chapter
      if (!shouldResume) return
      
      // 5. 检查空闲时长
      const idleDuration = Date.now() - monitorLastActivity.current
      const idleThreshold = monitorConfig.idleTimeoutMinutes * 60 * 1000
      if (idleDuration < idleThreshold) return
      
      // 6. 生成续传消息
      const resumeMessage = `<system_notification>
当前是自动续传任务，触发条件：card.json 中 target_chapters（${card.target_chapters}）> current_chapter（${card.current_chapter}）。

请继续写第${card.current_chapter + 1}章，一直写到第${card.target_chapters}章。

⚠️ 重要：每完成一章后，请立即更新 card.json 中的 current_chapter 数值。
   - 写完第 N 章后，将 current_chapter 设为 N
   - 当 current_chapter >= target_chapters 时，表示全部完成，监控将自动停止
</system_notification>`
      await sendMessage(threadId, resumeMessage)
      
      // 7. 更新活动时间
      monitorLastActivity.current = Date.now()
      
    } catch (error) {
      console.error('[Monitor] Error:', error)
      // 读取 card.json 失败不应中断监控
    }
  }, 30000)
  
  monitorCheckTimer.current = checkTimer
}

// 停止监控
function stopMonitoring() {
  if (monitorCheckTimer.current) {
    clearInterval(monitorCheckTimer.current)
    monitorCheckTimer.current = null
  }
  setMonitorEnabled(false)
}
```

## 5. 注意事项

1. **监控范围**：只对 `novel-master` Agent 的会话有效
2. **card.json 路径**：从会话的 workDir 读取 `book/[小说名称]/card.json`
3. **错误处理**：读取 card.json 失败时不应中断监控，记录日志即可
4. **性能影响**：30 秒轮询对性能影响可忽略
5. **多会话支持**：每个会话独立控制监控开关
6. **持久化**：刷新页面后监控状态重置（可选：保存到 localStorage）
7. **消息去重**：续传后重置空闲计时器，确保不会短时间内重复发送
8. **空闲判断**：基于 `thread.isLoading` 状态（来自 `useStream` hook）
9. **配置面板关闭**：点击"取消"或开始监控后自动关闭面板
10. **续传机制**：通过 `sendMessage()` 发起新 run，LangGraph checkpointer 自动加载历史

## 7. 测试场景

| 场景 | 预期行为 |
|------|---------|
| 点击监控按钮 | 弹出配置面板 |
