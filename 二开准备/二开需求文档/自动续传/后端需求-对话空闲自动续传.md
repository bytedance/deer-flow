# 后端需求 - 对话空闲自动续传

> Agent 对话过程中可能因各种原因中断（完成任务后停止、等待用户确认等），需要一种机制自动检测并续传任务，支持整夜挂机执行。

---

## 1. 业务需求

### 1.1 背景

用户在整夜挂机执行写正文流程时，Agent 会在某个环节完成后停止（认为当前回合结束），需要用户手动输入"继续"才能执行下一步。这导致无法真正实现全自动挂机。

### 1.2 核心需求

1. **持续监控**：系统需要持续监控当前正在执行的对话状态
2. **触发条件**：用户自定义判断条件（如"Agent 空闲超过 10 分钟" + "还有待完成的任务"）
3. **自动续传**：触发条件满足后，自动向当前对话发送续传消息
4. **可配置**：监控间隔、空闲阈值、触发条件、续传消息都由用户配置
5. **可控**：可以随时启用/禁用监控服务

### 1.3 使用场景

| 场景 | 说明 |
|------|------|
| 整夜写正文 | Agent 写一章完成后空闲，自动触发续传写下一章 |
| 长任务执行 | 多个步骤的任务，某步完成后等待确认，自动续传 |
| 自动重试 | 审核不通过需要修改时，自动触发 Reviser |

### 1.4 用户交互

用户需要能够：
1. 设置一个判断方法（返回 true 表示应该续传）
2. 设置一个消息生成方法（返回要发送的续传消息）
3. 启动/停止监控服务

---

## 2. 技术方案

### 2.1 架构设计

本项目使用 **LangGraph Server 模式**，对话状态通过 `AsyncSqlSaver`（checkpointer）持久化到 SQLite/PostgreSQL。每次发起新 run 时，checkpointer 自动加载历史消息。

```
前端 chatStore 定时器
  |
  |-- 每 30 秒检查 --► thread.isLoading === false（空闲）
  |                    card.json 还有未写章节
  |
  | 条件满足
  |--► sendMessage(threadId, "请继续写第 N 章")
  |       |
  |       |--► POST /api/threads/{id}/runs/stream
  |               body: { input: { messages: [{type: "human", content: ...}] } }
  |
  |       |--► LangGraph checkpointer 自动加载历史
  |       |--► Agent 基于完整历史继续执行
```

### 2.2 关键点分析

#### 2.2.1 如何检测 Agent 空闲

项目使用 `useStream` React hook（[hooks.ts:425-531](file:///c:/xiangmu/deer-flow/frontend/src/core/threads/hooks.ts#L425-L531)）管理流式响应，`thread.isLoading` 精确反映 LLM 是否正在输出：

```typescript
// frontend/src/components/workspace/input-box.tsx:153-159
status = thread.error ? "error" : thread.isLoading ? "streaming" : "ready"
```

- `isLoading === true` → LLM 正在流式输出
- `isLoading === false` → 空闲（run 已结束或尚未开始）

#### 2.2.2 如何发消息续传

项目已有 `sendMessage(threadId, message)` 函数（[hooks.ts:326-519](file:///c:/xiangmu/deer-flow/frontend/src/core/threads/hooks.ts#L326-L519)）：

```typescript
// 发起新 run，checkpointer 自动加载历史
await thread.submit(
  { messages: [{ type: "human", content: [{ type: "text", text }] }] },
  { threadId, streamSubgraphs: true, streamResumable: true }
);
```

每次调用 `sendMessage()` 就是向 LangGraph Server 发起 `POST /api/threads/{thread_id}/runs/stream`，与用户手动点击"发送"完全相同。

#### 2.2.3 为什么不需要后端 idleWatcher 服务

- 监控逻辑完全可以在前端 chatStore 中实现
- 前端已有 `thread.isLoading` 状态
- 前端已有 `sendMessage()` API
- 后端 checkpointer 自动管理对话历史
- 不需要后端主动发送消息到 WebSocket

### 2.3 需要后端支持的部分

#### 2.3.1 读取 card.json

项目已有 [filesystem API](file:///c:/xiangmu/deer-flow/backend/app/gateway/routers/filesystem.py)，支持读取 `.json` 文件。前端通过 `GET /api/threads/{thread_id}/file?path=card.json` 读取。

确保 card.json 在可访问路径（workDir/book/[小说名称]/card.json），前端可通过文件系统 API 读取。

#### 2.3.2 确保续传消息不会触发错误处理中间件

续传消息应该作为普通 HumanMessage 发送，不被 [llm_error_handling_middleware](file:///c:/xiangmu/deer-flow/backend/packages/harness/deerflow/agents/middlewares/llm_error_handling_middleware.py) 拦截。

确保续传消息格式正确，不会触发 [loop_detection_middleware](file:///c:/xiangmu/deer-flow/backend/packages/harness/deerflow/agents/middlewares/loop_detection_middleware.py)。

### 2.4 Agent 配合

Agent 的 systemPrompt 需要包含"收到'继续'后自动续传"的规则。可以通过以下方式实现：

1. 在 lead_agent 的 systemPrompt 中添加规则
2. 或通过 middleware 在续传消息前注入上下文

### 2.5 环境变量配置（可选）

```env
# .env
AUTO_RESUME_ENABLED=true
AUTO_RESUME_CHECK_INTERVAL=30000        # 检查间隔（毫秒），默认 30000
AUTO_RESUME_IDLE_THRESHOLD=600000       # 空闲阈值（毫秒），默认 600000（10 分钟）
AUTO_RESUME_AGENT_NAME=novel-master     # 只监控特定 Agent（可选）
```

## 3. 注意事项

1. **消息重复发送**：续传后重置空闲计时器，确保不会短时间内重复触发
2. **会话状态准确性**：依赖 `thread.isLoading` 判断空闲状态
3. **错误处理**：`sendMessage()` 失败时应有日志记录
4. **性能影响**：30 秒轮询对性能影响可忽略
5. **Agent 配合**：主 Agent 的 systemPrompt 需要包含"收到'继续'后自动续传"的规则
6. **多会话支持**：每个会话独立控制监控开关
7. **刷新页面**：监控状态重置（可选：保存到 localStorage）
