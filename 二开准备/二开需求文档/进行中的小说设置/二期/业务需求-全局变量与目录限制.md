# 二期需求文档：小说标签全局变量与目录限制（业务需求）

> 目标：将小说标签作为全局变量注入 Agent 提示词，并限制资源管理器只能查看当前小说目录

---

## 1. 需求概述

### 1.1 功能目标
1. **全局变量注入**：将当前对话的小说标签作为全局变量，注入到 Agent 的系统提示词中
2. **目录访问限制**：右侧资源管理器只能查看和操作当前小说标签对应的目录

### 1.2 业务背景
- 一期实现了小说标签的创建和分组展示
- 二期需要让 Agent "知道"当前正在编辑哪部小说
- 同时需要隔离不同小说的文件访问，避免误操作
- **注意**：二期技术实现方案待全局变量需求明确后另行设计

---

## 2. 业务流程设计

### 2.1 全局变量注入流程

#### 业务场景描述

当用户创建对话并选择了小说标签（如"小说A"）后：
1. 系统在后续的所有对话交互中，自动将"小说A"这个信息传递给 Agent
2. Agent 在处理用户请求时，能够明确知道当前操作的是"小说A"项目
3. Agent 生成的所有文件操作、代码编写等内容，都会围绕"小说A"展开

#### 提示词注入效果示例

```
你是一个专业的小说创作助手。

当前正在编辑的小说：小说A

请根据这部小说的上下文进行创作和修改。
所有文件操作都应该在 /mnt/user-data/workspace/book/小说A/ 目录下进行。

用户需求：帮我写第一章...
```

#### 变量传递链路（概念说明）

```
对话页面（读取 novel_tag）
    ↓
前端传递给后端
    ↓
后端读取 novel_tag
    ↓
注入到 Agent 提示词模板
    ↓
Agent 收到包含小说信息的提示词
```

### 2.2 资源管理器目录限制流程

#### 业务场景描述

当用户打开一个已设置小说标签的对话：
1. 右侧资源管理器自动定位到该小说的专属目录
2. 用户只能浏览和操作该目录下的文件
3. 无法访问其他小说目录，避免误操作

#### 目录结构示例

```
/mnt/user-data/workspace/
└── book/
    ├── 小说A/          ← 当前对话的资源管理器根目录
    │   ├── chapters/
    │   │   ├── chapter1.md
    │   │   └── chapter2.md
    │   ├── characters/
    │   │   └── character_list.md
    │   └── outline.md
    ├── 小说B/          ← 不可访问
    └── 小说C/          ← 不可访问
```

#### 访问控制规则

| 场景 | 系统行为 |
|------|---------|
| 对话已设置 novel_tag | 资源管理器自动定位到对应小说目录 |
| 对话未设置 novel_tag | 资源管理器显示提示"请先选择小说标签"或禁用 |
| 用户尝试访问其他目录 | 系统拒绝访问，显示权限错误 |

## 3. 业务逻辑详细说明

### 3.1 全局变量管理

#### 3.1.1 变量定义

| 属性 | 说明 |
|------|------|
| 变量名 | novel_tag |
| 值类型 | 字符串（如"小说A"）或 undefined |
| 作用域 | 当前对话（不同对话可以有不同值） |
| 生命周期 | 对话创建后持久化，不可修改 |
| 用途 | 告知 Agent 当前正在编辑的小说项目 |

#### 3.1.2 变量读取时机

1. **对话加载时**：系统从对话的 metadata 中读取 novel_tag
2. **发送消息前**：系统将 novel_tag 包含在请求上下文中
3. **Agent 处理时**：后端将 novel_tag 注入到提示词中

#### 3.1.3 使用场景

**场景 1：文件操作**
- Agent 创建文件时，自动保存到正确的小说目录
- Agent 读取文件时，只读取当前小说目录下的文件

**场景 2：内容创作**
- Agent 知道当前编辑的是哪部小说
- 可以参考该小说的历史内容和设定

**场景 3：代码生成**
- 生成的代码文件保存到正确的目录结构
- 避免文件散落在错误的目录中

### 3.2 资源管理器限制逻辑

#### 3.2.1 目录隔离原则

每个小说项目有独立的目录空间：
- 防止不同小说的文件混淆
- 提高文件管理的清晰度
- 降低误操作风险

#### 3.2.2 用户体验

**有小说标签的对话**：
```
┌────────────────────────────────────┐
│ 📂 资源管理器                       │
│                                     │
│ 📁 小说A                            │
│   ├── 📁 chapters/                  │
│   │   ├── chapter1.md               │
│   │   └── chapter2.md               │
│   ├── 📁 characters/                │
│   └── 📄 outline.md                 │
│                                     │
│ [新建文件] [刷新] [下载]            │
└────────────────────────────────────┘
```

**没有小说标签的对话**：
```
┌────────────────────────────────────┐
│ 📂 资源管理器                       │
│                                     │
│ ⚠️ 此对话未设置小说标签             │
│ 请先新建对话并选择小说标签          │
│                                     │
│ [新建对话]                          │
└────────────────────────────────────┘
```

## 4. 关键设计决策

### 4.1 全局变量概念澄清

**重要说明**：这里的"全局变量"实际上应该称为"对话上下文变量"或"运行时变量"：

- ❌ **不是跨对话共享的全局变量**（所有对话共用一个值）
- ✅ **是对话级别的变量**（每个对话有自己的值）

每个对话的 novel_tag 是独立的，互不影响。

### 4.2 目录限制层级

建议采用**双重限制**方案：

| 层级 | 作用 |
|------|------|
| 前端 UI 限制 | 资源管理器组件只显示允许的目录，提供良好用户体验 |
| 后端 API 限制 | 文件操作 API 验证访问路径，保证安全性 |

前端限制用于用户体验，后端限制用于安全保障。

## 6. 业务场景示例

### 场景 1：用户创建小说对话

1. 用户点击"新建对话"
2. 选择小说标签："三体创作"
3. 输入消息："帮我写第一章"
4. 系统提示词中注入"当前小说：三体创作"
5. Agent 回复时知道当前是"三体创作"项目
6. 所有文件操作都在 `/mnt/user-data/workspace/book/三体创作/` 下进行

### 场景 2：用户在对话中浏览文件

1. 用户打开已有的"三体创作"对话
2. 点击右侧"文件浏览器"
3. 文件浏览器只显示"三体创作"目录下的文件
4. 用户可以查看、编辑、创建该目录下的文件
5. 无法访问其他小说目录（如"流浪地球"）

### 场景 3：未设置标签的对话

1. 用户创建对话时未选择小说标签
2. 对话的 metadata 中 novel_tag 为空
3. Agent 提示词中不包含小说信息（作为通用助手）
4. 资源管理器显示"请先设置小说标签"

## 8. 后续扩展考虑

### 8.1 多标签支持
- 一期只支持单标签
- 未来可能支持一个对话关联多部小说
- 需要改为 `novel_tags: string[]` 数组

### 8.2 标签层级
- 当前标签是扁平的
- 未来可能需要分类/子标签
- 例如："科幻/三体"、"武侠/射雕"

### 8.3 标签权限
- 当前所有标签对所有用户可见
- 未来可能需要私有标签
- 基于用户 ID 的标签隔离

---

## 9. 风险与挑战

### 9.1 业务风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 用户选错标签 | 文件位置错误 | 一期锁定标签不可修改，提供二次确认 |
| 标签命名不规范 | 管理混乱 | 由 Agent 创建目录，保证命名规范 |

### 9.2 技术风险（待评估）

| 风险 | 说明 |
|------|------|
| 提示词长度增加 | 注入变量可能增加 token 消耗 |
| 目录限制绕过 | 需要在后端强制验证，不能仅依赖前端 |
