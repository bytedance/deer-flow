# Deer Desktop V1 Spec

Date: 2026-04-08
Status: Draft

## Goal

Build a desktop version of DeerFlow for single-user local usage by wrapping the existing DeerFlow web/runtime architecture in Electron, without changing DeerFlow's existing LLM provider model.

## Product Scope

Deer Desktop V1 is:
- single-user
- single-instance
- single data directory
- local-first
- based on the existing DeerFlow frontend + gateway + runtime stack

Deer Desktop V1 does not:
- introduce a new model provider layer
- replace DeerFlow providers with a proxy layer
- introduce multi-workspace switching
- replace the current DeerFlow service architecture with embedded DeerFlowClient

## Architecture Decision

V1 uses the existing DeerFlow service-style architecture inside an Electron shell.

### Included runtime pieces
- Electron shell
- DeerFlow frontend
- DeerFlow gateway
- DeerFlow runtime (LangGraph server or gateway-mode runtime)
- existing DeerFlow model/provider system

### Explicitly deferred
- embedded DeerFlowClient runtime as the primary backend architecture
- provider-level rewrites
- workspace switching
- deep sandbox refactors

## User Data Directory

The app uses a single per-user application data directory.

Preferred implementation:
- use Electron `app.getPath("userData")`

Typical locations:
- macOS: `~/Library/Application Support/Deer`
- Windows: `%APPDATA%\\Deer`

Directory layout:

```text
<userData>/
├── config.yaml
├── extensions_config.json
├── state/
│   ├── preferences.json
│   └── desktop.json
├── outputs/
├── runtime/
└── skills/
    └── custom/
```

## Configuration Model

### config.yaml
Generated by the Electron main process.

Responsibilities:
- define DeerFlow models using DeerFlow's native provider system
- configure sandbox mode
- configure skills path
- reference secrets through environment variables

V1 principles:
- do not persist API keys in plaintext config
- continue using DeerFlow's existing provider classes
- default to local sandbox mode

### extensions_config.json
Generated and updated by the Electron main process.

Responsibilities:
- MCP server configuration
- skill enabled state

V1 default:
- include filesystem MCP
- keep skill state here

## Runtime Mode

V1 is local-first and desktop-first, but does not remove DeerFlow's existing sandbox abstraction.

### Runtime choice
- use `deerflow.sandbox.local:LocalSandboxProvider`

### Product behavior
- local file access is treated as expected behavior in desktop mode
- Docker assumptions should be removed from desktop-facing behavior
- final outputs should be easy for the user to locate in the desktop data area

### Non-goal for V1
- redesigning DeerFlow's `/mnt/user-data/{workspace,uploads,outputs}` thread model

Instead, V1 should:
- keep DeerFlow's internal thread/runtime model intact where possible
- relocate runtime/thread data under `<userData>/runtime`
- make outputs visible and easy to access from the desktop app

## Settings IA

Desktop UI requires a Settings area beyond chat.

### Sections
- General
- Models & API Keys
- Tools & MCP
- Skills
- Advanced

### General
Show:
- data directory
- outputs directory
- runtime directory
- runtime/backend status
- app version

### Models & API Keys
Must support:
- selecting default model
- configuring API keys for DeerFlow-supported providers
- showing configured/not configured status per provider

Initial provider targets:
- OpenAI
- Anthropic
- DeepSeek
- other DeerFlow-supported providers as needed

Storage rules:
- secrets stored via Electron `safeStorage`
- non-sensitive settings stored in state JSON
- secrets injected into DeerFlow child process via environment variables
- secrets must not be stored in plaintext in `config.yaml`

### Tools & MCP
Must support:
- viewing MCP server list
- enabling/disabling MCP servers
- showing mounted local directories where relevant

### Skills
Must support:
- listing skills
- enabling/disabling skills
- installing custom skills later if already supported by DeerFlow

### Advanced
Initial candidates:
- allow host bash toggle
- open data directory
- rebuild config
- restart runtime

## Frontend Changes

The existing settings dialog should be extended with a new "Models & API Keys" section.

High-level changes:
- add settings section entry
- add a model settings page
- keep cosmetic/UI preferences in frontend local settings
- move system-level settings to desktop-managed storage/API

The frontend should not persist raw API keys in localStorage.

## Electron Responsibilities

The Electron main process is the control plane for the desktop app.

### Required responsibilities
- resolve data paths
- initialize user data directories
- generate `config.yaml`
- generate `extensions_config.json`
- securely store API keys
- inject API keys as env vars when starting DeerFlow runtime
- start/stop DeerFlow processes
- expose IPC for settings and secret management

### Suggested modules
```text
desktop/main/
├── index.ts
├── paths.ts
├── config.ts
├── secrets.ts
├── processes.ts
└── ipc.ts
```

## Backend Responsibilities

V1 should minimize backend churn.

### Required backend adjustments
1. Respect Electron-provided config paths:
   - `DEER_FLOW_CONFIG_PATH`
   - `DEER_FLOW_EXTENSIONS_CONFIG_PATH`

2. Add desktop-mode prompt/context guidance so the lead agent understands:
   - it is running in a single-user desktop environment
   - local file operations are expected when requested
   - outputs should be surfaced to desktop-visible locations
   - Docker/container assumptions should be avoided

3. Move DeerFlow runtime/thread data under the desktop user data root instead of repo-local defaults where needed.

4. Optionally enrich model metadata for the frontend with non-secret provider requirements such as required env var names.

## API / IPC Requirements

Desktop-facing settings need a thin control interface.

Minimum required capabilities:
- get desktop settings
- update desktop settings
- get API key configured status
- save API key
- delete API key
- restart runtime
- get app paths

Whether these are exposed through Electron IPC only or through a thin local bridge is an implementation detail for the next planning phase.

## Outputs Behavior

V1 should preserve DeerFlow's existing artifact/thread model where practical.

Desired behavior:
- DeerFlow internal runtime files live under `<userData>/runtime`
- user-visible final outputs are easy to access from the desktop app
- settings and UI should expose the outputs directory

V1 should avoid a large artifact protocol rewrite.

## Non-Goals

The following are explicitly out of scope for V1:
- multi-workspace support
- custom proxy-based provider layer
- replacing DeerFlow runtime with DeerFlowClient as the primary architecture
- major sandbox abstraction rewrite
- large-scale output/artifact pipeline redesign

## P0 Implementation Priorities

1. Electron-managed user data directory
2. Generated `config.yaml` and `extensions_config.json`
3. Secure API key storage and env injection
4. Stable DeerFlow process startup/shutdown
5. Settings page with Models & API Keys
6. Desktop-mode prompt adjustments
7. Runtime/thread data rooted under desktop user data

## P1 Priorities

1. Better outputs surfacing
2. Open data directory actions
3. Allow host bash toggle
4. Non-secret provider metadata for model settings UI
5. Runtime rebuild/restart actions in settings

## Current Decision Summary

Chosen direction for V1:
- keep existing DeerFlow frontend/backend/runtime architecture
- wrap it in Electron
- use a single per-user app data directory
- do not change provider architecture
- add desktop settings with API key management
- treat local operation as the default desktop behavior

Deferred to later versions:
- embedded DeerFlowClient architecture
- deeper local-workspace/product-workspace redesign
- multi-workspace support
